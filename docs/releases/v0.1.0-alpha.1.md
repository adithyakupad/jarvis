# JARVIS v0.1.0-alpha.1

JARVIS is a local, text-first control layer for coding agents. It gives each repository a persistent project workspace, asks a selected provider to inspect and propose work, seals approval to an exact proposal revision, records execution evidence, and independently verifies supported test scripts.

This is a developer-oriented public alpha tested on macOS. Begin with a disposable Git repository whose changes you can inspect and recover.

## Verified workflow

The release workflow is: add an existing local repository, describe its stable purpose, submit an immediate coding task, review a repository-grounded proposal, approve the current revision, observe live persisted progress, review changed paths and independent validation, restore a structured **Where we left off** handoff, and restart JARVIS without losing completed history.

Codex planning and execution are implemented and live-tested. The Claude Code adapter implements planning, execution, session continuity, and constrained local permissions, but Claude Code has not been live-tested for this release. JARVIS never silently switches providers; an unavailable selected provider fails explicitly.

## Install and start

Prerequisites are Git, Node.js 22.12 or newer, npm, and an installed and authenticated provider.

```bash
git clone https://github.com/adithyakupad/jarvis.git
cd jarvis
npm install
npm run jarvis
```

Keep the terminal open and open the single localhost URL JARVIS prints. Press `Ctrl+C` to stop it. The production frontend and API share one loopback-only Fastify origin. To isolate local state, set `JARVIS_DATA_DIR` before startup.

## Safety boundaries

- Planning is read-only; repository edits require approval of the exact current proposal revision.
- Repository paths are canonicalized and loaded from persisted project state, not supplied during execution.
- Provider selection is explicit, with no mock or cross-provider fallback.
- JARVIS records fresh repository evidence before and after execution.
- JARVIS never automatically resets, cleans, stashes, commits, pushes, or rolls back changes.
- A compatible duplicate startup reuses the existing URL. Incompatible or unknown processes are reported and never terminated automatically.
- Active in-process work interrupted by restart is marked interrupted and is not rerun automatically.

## Independent validation

For JavaScript and TypeScript repositories with a supported, non-placeholder `package.json` test script, JARVIS detects npm, pnpm, Yarn, or Bun and runs the test command independently after provider execution. The observed exit code and bounded output are authoritative; provider-written claims that tests passed are not.

Repository test scripts are trusted project code. They can execute arbitrary project behavior, including child processes, so onboard only repositories you trust. Unsupported repository types report that automated validation was not run.

## Early Reverse Context

Each terminal run updates one structured handoff for its project. JARVIS preserves deterministic run, repository, file, and validation facts; labels user corrections separately; classifies model statements as inferred or unresolved; and marks the handoff potentially stale when the repository fingerprint changes. The server injects the current handoff into the next planning request so prior decisions and verified outcomes do not need to be re-entered.

Users can correct narrative project state without rewriting Git, validation, changed-file, provider-session, or approval evidence. A recommended next action fills the task box but never submits, approves, or executes work. Handoffs are project context, not authorization, and are not general personal or cross-project memory.

## Upgrade and restart note

Stop any older JARVIS terminal with `Ctrl+C`, run `npm run jarvis` from this release, and reload the printed URL. The browser verifies API schema and build compatibility before enabling project actions. Servers without `/api/health` are treated as incompatible. Do not delete locks or terminate unknown processes blindly; follow the PID and port diagnostics printed by JARVIS.

## Known limitations

- Release-verified on macOS only; other operating systems have not been tested for this release.
- Local terminal application with no desktop wrapper, cloud service, or durable external worker.
- Restart interrupts active work; files may already have changed and there is no automatic rollback.
- No automatic commit or push. Users must review every proposal and resulting change.
- Independent validation supports JavaScript/TypeScript `package.json` test scripts only.
- Claude Code remains implemented but unverified in this release environment.
- Chronological activity may contain technical provider and repository event details.
- Handoff narrative can be incomplete; deterministic evidence remains authoritative, and non-Git freshness is conservative.
- Reverse Context is project-scoped only. There is no personal memory, embeddings, cross-project memory, or background monitoring.
- Provider and model output can be incorrect even when a process completes successfully.
