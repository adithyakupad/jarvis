# JARVIS v0.2.0-alpha.1

**Release theme:** JARVIS now remembers where your project stands.

## Release overview

This release candidate combines structured project handoffs and automatic cross-run planning context with the Liquid Stark visual redesign. JARVIS retains one bounded, user-correctable understanding of each project while preserving raw run history as evidence.

Codex execution and handoff generation are implemented and live-tested. Claude Code execution and handoff support are implemented but not release-live-tested. JARVIS never silently falls back to another provider.

## Structured project handoffs

After a terminal run, JARVIS asynchronously generates **Where we left off** with the current objective and status, last meaningful action and outcome, changed files, independent validation, repository condition, blockers, decisions, constraints, and a recommended next action. Handoff generation does not reopen a completed execution or downgrade its terminal result.

Future planning automatically receives the selected project’s current handoff. Context is bounded by project and cannot cross project boundaries.

## Evidence boundaries and corrections

Git, run, changed-file, approval, and validation facts are deterministic JARVIS evidence. Model-derived statements remain classified as inferred or unresolved. Users may correct only narrative project state; saved corrections are classified `user-provided`, remain auditable, and take precedence over model inference. Corrections cannot alter source runs, Git evidence, provider sessions, approvals, or validation.

## Repository freshness

JARVIS compares the saved handoff fingerprint with the canonical repository’s current Git HEAD, status, dirty tracked content, visible untracked content, and relevant configuration. A mismatch marks the handoff potentially stale and displays a prominent warning. Planning then performs fresh inspection rather than silently trusting stored state. Repository-unavailable and non-Git states remain conservative.

## Liquid Stark redesign

The new interface provides a compact system bar, project/provider/runtime status, structured handoff card, read-only command composer, explicit approval boundary, authoritative validation, chronological SSE activity, and keyboard-accessible technical disclosures. It includes responsive layouts, visible focus, reduced-motion and forced-colors support, and an opaque fallback when backdrop filtering is unavailable.

## Independent validation

Provider narrative is not authoritative. JARVIS independently detects and runs supported repository tests after execution and displays the command, status, exit code, duration, stdout, and stderr. Unsupported, unavailable, failed, timed-out, and interrupted outcomes remain explicit.

## Installation and startup

Requirements: Node.js 22.12 or newer, npm, Git, and an installed/authenticated provider.

```bash
git clone https://github.com/adithyakupad/jarvis.git
cd jarvis
npm install
npm run jarvis
```

Open the one authoritative loopback URL printed by JARVIS. Stop the owned process with `Ctrl+C`.

## Upgrade behavior

Existing alpha databases are upgraded in place by SQLite migration 7, which adds the structured handoff, evidence, freshness, diagnostics, and correction records required by Reverse Context. Back up valuable local state before upgrading. API schema remains version 2; the visual integration does not change the API contract.

## Limitations

- This is developer-oriented pre-release software.
- Codex is the release-live-tested provider; Claude Code is implemented but not release-live-tested.
- Handoffs summarize project state but are not general memory and do not replace raw evidence.
- Non-Git repositories cannot receive a strong freshness guarantee.
- There is no durable background worker; stopping during work interrupts the in-process operation.
- macOS is the release-verified operating system.

## Safety and approval boundaries

Planning and handoff injection are read-only. A handoff provides context, never authorization. **Use recommended next step** only fills the task composer. Execution begins only after the user approves the exact current proposal revision. JARVIS does not commit, push, reset, clean, or silently change providers.

Intended Git tag: `v0.2.0-alpha.1`. Release preparation does not create or push the tag.
